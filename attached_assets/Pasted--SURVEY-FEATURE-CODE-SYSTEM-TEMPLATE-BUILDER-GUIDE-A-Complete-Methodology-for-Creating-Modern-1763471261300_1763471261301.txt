## üèóÔ∏è THE BUILD PROCESS - STEP BY STEP

### PHASE 1: ANALYSIS & ORGANIZATION (Week 1)

#### Step 1.1: Extract Current Nomenclature
Review all existing feature codes and create an inventory:

**Create a spreadsheet with:**
- Current Code (as used in field)
- Full Description
- Feature Type (Point, Line, Polygon)
- Current CAD Layer
- Frequency of Use (Daily, Weekly, Monthly, Rare)
- Notes/Issues

**Example:**
```
| Current Code | Description | Type | Layer | Frequency | Notes |
|--------------|-------------|------|-------|-----------|-------|
| SDMH | Storm Drain Manhole | Point | EX-SD-struct | Daily | Need invert elev |
| WV | Water Valve | Point | EX-WATER | Daily | Often missing depth |
| CB | Catch Basin | Point | EX-SD-struct | Daily | Same as SDCB? |
```

#### Step 1.2: Identify Categories
Group features into logical categories:

**Standard Categories:**
1. Survey Control (benchmarks, control points, GPS stations)
2. Storm Drain System (manholes, catch basins, lines)
3. Sanitary Sewer System (manholes, cleanouts, lines)
4. Water System (valves, hydrants, meters, lines)
5. Recycled Water (if applicable)
6. Irrigation System (if applicable)
7. Gas System (valves, meters, lines)
8. Electric System (poles, vaults, transformers, lines)
9. Lighting System (street lights, poles)
10. Telecommunications (phone, fiber, cable)
11. Generic/Unknown Utilities
12. Topographic Features (banks, curbs, edges, trees)
13. Site Improvements (benches, signs, gates, etc.)
14. Structures (buildings, walls, doors)
15. Special Features (project-specific items)

**Customize based on your work!**

#### Step 1.3: Define Nomenclature Rules
Establish a consistent naming convention:

**Option A: Prefix-Based Hierarchical**
```
[CATEGORY]-[FEATURE][MODIFIER]

Examples:
SD-MH = Storm Drain - Manhole
W-V = Water - Valve
EL-XFMR = Electric - Transformer
```

**Option B: Compound Abbreviation**
```
[CATEGORY][FEATURE]

Examples:
SDMH = Storm Drain Manhole
WV = Water Valve
ELXFMR = Electric Transformer
```

**Option C: Hybrid (Support Both)**
```
Legacy in field: SDMH
Database stores: SD-MH
Automatic translation layer
```

**Document all abbreviations:**
- Category prefixes (SD, SS, W, G, EL, TEL, etc.)
- Component types (MH, CB, V, BOX, RISER, etc.)
- Modifiers (OH=Overhead, UG=Underground, etc.)

### PHASE 2: ATTRIBUTE DESIGN (Week 1-2)

#### Step 2.1: Define Attribute Requirements by Feature Type

**For Each Feature, Identify:**
1. What data MUST be collected (required fields)
2. What data SHOULD be collected (optional but recommended)
3. What data is NICE to have (optional)

**Standard Attribute Patterns:**

**Utility Structures (Manholes, Vaults, Boxes):**
```
Required:
- Rim/Top Elevation (numeric, 2 decimals)
- Material (dropdown: Concrete, Precast, Block, Brick, Plastic, Fiberglass)
- Size/Diameter (text or numeric)

Optional:
- Invert Elevation (numeric, 2 decimals)
- Condition (dropdown: Excellent, Good, Fair, Poor, Failed)
- Manufacturer (text)
- Install Date (date)
- Owner (dropdown: City, County, Private, HOA)
- Notes (text, 200 char max)
```

**Valves:**
```
Required:
- Valve Type (dropdown: Gate, Butterfly, Ball, Check, etc.)
- Size (numeric, 1 decimal)

Optional:
- Depth to Valve (numeric, 1 decimal)
- Turns to Close (numeric, integer)
- Condition (dropdown)
- Last Exercised Date (date)
- Notes (text)
```

**Utility Lines:**
```
Required:
- Material (dropdown: PVC, DIP, AC, CMP, RCP, etc.)
- Size/Diameter (numeric)

Optional:
- Depth (numeric, 1 decimal)
- Pressure Rating (numeric)
- Install Year (numeric, 4 digits)
- Condition (dropdown)
- Notes (text)
```

**Topographic Features:**
```
Required:
- Elevation (auto from point)

Optional:
- Feature Type (dropdown based on code)
- Description (text)
- Condition (dropdown)
```

#### Step 2.3: Define Data Types and Validation

**Data Type Options:**
- **String/Text**: Free-form text (set max length: 20, 50, 100, 200 chars)
- **Numeric**: Numbers with decimal precision (0, 1, 2, 3 decimals)
- **PickList/Dropdown**: Standardized values from predefined list
- **Date**: Date picker (MM/DD/YYYY or other format)
- **Boolean**: Yes/No or True/False checkbox
- **Coordinate**: Northing/Easting pairs

**Validation Rules:**
- Required vs. Optional
- Min/Max values for numerics
- Character limits for text
- Allowed values for dropdowns
- Format rules (dates, coordinates)

    </Attributes>
  </Feature>

### PHASE 5: DATABASE INTEGRATION DESIGN (Week 3)

#### Step 5.1: Design Database Schema

**If using PostgreSQL/PostGIS:**

**Core Tables:**
```sql
-- Feature code definitions
CREATE TABLE feature_codes (
    code_id UUID PRIMARY KEY,
    feature_code VARCHAR(50) UNIQUE NOT NULL,
    feature_name VARCHAR(200) NOT NULL,
    category VARCHAR(50) NOT NULL,
    geometry_type VARCHAR(20) NOT NULL,
    cad_layer VARCHAR(100),
    cad_color INTEGER,
    is_active BOOLEAN DEFAULT true,
    attributes_schema JSONB
);

-- Projects
CREATE TABLE projects (
    project_id UUID PRIMARY KEY,
    project_number VARCHAR(50) UNIQUE NOT NULL,
    project_name VARCHAR(200) NOT NULL,
    coordinate_system VARCHAR(100),
    project_bounds GEOMETRY(POLYGON)
);

-- Survey sessions
CREATE TABLE survey_sessions (
    session_id UUID PRIMARY KEY,
    project_id UUID REFERENCES projects,
    session_date DATE NOT NULL,
    crew_chief VARCHAR(100),
    data_collector_file VARCHAR(200)
);

-- Points
CREATE TABLE points (
    point_id UUID PRIMARY KEY,
    project_id UUID REFERENCES projects,
    session_id UUID REFERENCES survey_sessions,
    feature_code VARCHAR(50) REFERENCES feature_codes(feature_code),
    point_number VARCHAR(50),
    geometry GEOMETRY(POINTZ, SRID) NOT NULL,
    attributes JSONB,  -- All feature-specific attributes
    collection_datetime TIMESTAMP,
    created_by VARCHAR(100)
);

-- Lines
CREATE TABLE lines (
    line_id UUID PRIMARY KEY,
    project_id UUID REFERENCES projects,
    session_id UUID REFERENCES survey_sessions,
    feature_code VARCHAR(50) REFERENCES feature_codes(feature_code),
    geometry GEOMETRY(LINESTRINGZ, SRID) NOT NULL,
    attributes JSONB,
    created_by VARCHAR(100)
);

-- Polygons
CREATE TABLE polygons (
    polygon_id UUID PRIMARY KEY,
    project_id UUID REFERENCES projects,
    session_id UUID REFERENCES survey_sessions,
    feature_code VARCHAR(50) REFERENCES feature_codes(feature_code),
    geometry GEOMETRY(POLYGONZ, SRID) NOT NULL,
    attributes JSONB,
    created_by VARCHAR(100)
);

-- Spatial indexes
CREATE INDEX idx_points_geom ON points USING GIST(geometry);
CREATE INDEX idx_lines_geom ON lines USING GIST(geometry);
CREATE INDEX idx_polygons_geom ON polygons USING GIST(geometry);
```

**If using SQL Server with Spatial:**
Similar structure but use `geometry` data type instead of PostGIS `GEOMETRY`.

**If using Esri Geodatabase:**
Create feature classes with same logical structure.

## üéØ CRITICAL DESIGN DECISIONS

### Decision 1: Code Format Strategy

**Option A: Maintain Legacy Codes**
- Field crews use familiar codes (SDMH, WV)
- Database stores same format
- PRO: No translation needed, crew familiarity
- CON: Less organized for querying

**Option B: Force New Hierarchical Codes**
- Field crews must learn new codes (SD-MH, W-V)
- Database stores same format
- PRO: Clean, organized, queryable
- CON: Training burden, resistance to change

**Option C: Dual-Code System (RECOMMENDED)**
- Field crews use legacy codes (SDMH, WV)
- Database auto-translates to hierarchical (SD-MH, W-V)
- Bidirectional mapping maintained
- PRO: Best of both worlds, no crew disruption
- CON: Slightly more complex import/export

**Recommendation: Choose Option C for smoothest transition**

### Decision 2: Attribute Collection Strategy

**Required vs. Optional Fields**

**Aggressive Approach:**
- Make most fields required
- Forces complete data collection
- PRO: Complete dataset
- CON: Crew frustration, slower collection

**Lenient Approach:**
- Make few fields required
- Crews can skip most attributes
- PRO: Fast collection
- CON: Incomplete data

**Balanced Approach (RECOMMENDED):**
- Require only critical fields (elevation, feature type)
- Recommend important fields (material, size, condition)
- Optional for nice-to-have fields (install date, manufacturer)
- PRO: Quality data without slowing crews
- CON: Some missing data inevitable

### Decision 3: Backward Compatibility

**How to Handle Legacy Projects?**

**Option A: Dual System**
- Keep old system for old projects
- Use new system for new projects
- Never migrate old data
- PRO: No migration effort
- CON: Two systems to maintain

**Option B: Hard Cutover**
- Set a date, switch completely
- Old projects stay in old format
- All new work uses new system
- PRO: Clean break
- CON: Can't easily compare old/new projects

**Option C: Gradual Migration (RECOMMENDED)**
- Deploy new system for new projects
- Migrate old projects on as-needed basis
- Maintain import for old format indefinitely
- PRO: Flexibility, comprehensive dataset eventually
- CON: Migration work over time

### File Format Specifications:

**Trimble FXL (XML) Structure:**
```xml
<?xml version="1.0" encoding="UTF-8"?>
<FeatureLibrary version="7.0" xmlns="http://www.trimble.com/schema/SurveyFeatureLibrary">
  <LibraryInfo>
    <n>Library Name</n>
    <Description>Library description</Description>
    <Version>1.0</Version>
    <Date>YYYY-MM-DD</Date>
    <Company>Company Name</Company>
    <CoordinateSystem>Coordinate system description</CoordinateSystem>
  </LibraryInfo>

  <!-- Point Features -->
  <Feature>
    <Code>CODE</Code>
    <n>Full Name</n>
    <Description>Description</Description>
    <GeometryType>Point</GeometryType>
    <Layer>LAYER_NAME</Layer>
    <Color>ColorName</Color>
    <PointStyle>Circle|Square|Triangle|Diamond|X|Star</PointStyle>
    <PointSize>1.0-10.0</PointSize>
    <Attributes>
      <!-- Attribute definitions -->
    </Attributes>
  </Feature>

  <!-- Line Features -->
  <LineCode>
    <Code>CODE</Code>
    <n>Full Name</n>
    <Description>Description</Description>
    <GeometryType>Line</GeometryType>
    <Layer>LAYER_NAME</Layer>
    <Color>ColorName</Color>
    <LineType>Continuous|Dashed|DashDot|Dotted</LineType>
    <LineWeight>0.10-1.00</LineWeight>
    <Attributes>
      <!-- Attribute definitions -->
    </Attributes>
  </LineCode>

  <!-- Polygon Features -->
  <PolygonCode>
    <Code>CODE</Code>
    <n>Full Name</n>
    <Description>Description</Description>
    <GeometryType>Polygon</GeometryType>
    <Layer>LAYER_NAME</Layer>
    <Color>ColorName</Color>
    <FillPattern>Solid|Hatch|etc</FillPattern>
    <Attributes>
      <!-- Attribute definitions -->
    </Attributes>
  </PolygonCode>
</FeatureLibrary>
```

**Attribute Definition Structure:**

## üí° PRO TIPS & LESSONS LEARNED

### Start Small:
- Don't try to create 300 features on day one
- Start with 30-50 most common features
- Add more features over time as needed

### Involve Field Crews Early:
- Get their input during design
- Have them test early versions
- Listen to their feedback
- Make them champions of the new system

### Prioritize Data Quality:
- Better to have complete data on fewer attributes
- Than incomplete data on many attributes
- Start with critical fields only
- Add more as crews get comfortable

### Plan for the Unknown:
- Include generic codes (UTIL, UNKNOWN, MISC)
- Allow free-text notes field always
- Don't force codes for everything

### Make it Easy:
- Use familiar abbreviations where possible
- Keep dropdown lists short (5-10 items max)
- Make common values at top of list
- Use logical display order for attributes

### Documentation Matters:
- Field crews won't read 100-page manuals
- Create quick reference cards
- Use visual aids and examples
- Make help easily accessible

### Test Thoroughly:
- Test import to data collector
- Test collection workflow
- Test export from data collector
- Test import to database/CAD
- Test full round-trip

### Plan for Legacy:
- You'll be supporting old and new for years
- Don't delete old codes immediately
- Maintain compatibility where possible
- Plan migration thoughtfully
